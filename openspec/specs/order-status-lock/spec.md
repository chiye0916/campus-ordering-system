## ADDED Requirements

### Requirement: Redis Lock Protects Order Status Transitions
The system SHALL acquire a Redis distributed lock before executing order status transition logic for payment, cancellation, or completion.

#### Scenario: Payment obtains order status lock
- **WHEN** a user requests payment for an order
- **THEN** the system MUST acquire `lock:order:status:{orderId}` before creating payment records or updating order status

#### Scenario: Cancellation obtains order status lock
- **WHEN** a user requests cancellation for an order
- **THEN** the system MUST acquire `lock:order:status:{orderId}` before updating order status

#### Scenario: Completion obtains order status lock
- **WHEN** an administrator requests completion for an order
- **THEN** the system MUST acquire `lock:order:status:{orderId}` before updating order status

### Requirement: Lock Acquisition Failure Rejects Transition
The system SHALL reject an order status transition when the order status lock cannot be acquired.

#### Scenario: Concurrent request fails to acquire lock
- **WHEN** another request already holds `lock:order:status:{orderId}`
- **THEN** the system MUST reject the new status transition with a business error asking the caller to retry later

#### Scenario: Rejected transition does not update order
- **WHEN** a status transition request cannot acquire the lock
- **THEN** the system MUST NOT update the order status

### Requirement: Lock Ownership Is Safely Released
The system SHALL release Redis locks only when the stored lock value matches the value generated by the current request.

#### Scenario: Owner releases lock
- **WHEN** the request finishes after acquiring a lock
- **THEN** the system MUST release the lock with an atomic compare-and-delete operation

#### Scenario: Non-owner does not release lock
- **WHEN** the lock value in Redis differs from the current request's lock value
- **THEN** the system MUST NOT delete the Redis lock key

### Requirement: Database Status Conditions Remain Final Guard
The system SHALL keep database status-condition updates as the final consistency guard for order status transitions.

#### Scenario: Payment final guard remains
- **WHEN** the system updates an order from pending payment to paid
- **THEN** the update MUST require the order to still be pending payment in the database

#### Scenario: Completion final guard remains
- **WHEN** the system updates an order from paid to completed
- **THEN** the update MUST require the order to still be paid in the database

#### Scenario: Cancellation final guard applies
- **WHEN** the system updates an order to cancelled
- **THEN** the update MUST require the order to still be pending payment in the database

#### Scenario: Paid order cannot be directly cancelled
- **WHEN** a user requests cancellation for a paid order
- **THEN** the system MUST reject the cancellation and MUST NOT update the order to cancelled
